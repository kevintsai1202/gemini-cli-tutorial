## 技巧 9：使用 GEMINI CLI 進行系統故障排除和配置

**快速用例：** 您可以在程式碼專案之外運行 Gemini CLI，以協助處理一般系統任務——將其視為您的作業系統的智慧助理。例如，如果您的 shell 表現異常，您可以在主目錄中開啟 Gemini 並詢問：「修復我的 `.bashrc` 檔案，它有錯誤。」Gemini 隨後可以為您開啟和編輯您的設定檔。

此技巧強調 Gemini CLI 不僅僅適用於編碼專案——它是您整個開發環境的 AI 助手。許多用戶已使用 Gemini 自訂其開發設定或修復其機器上的問題：

*   **編輯點檔案：** 您可以透過引用您的 shell 配置（`.bashrc` 或 `.zshrc`）（`@~/.bashrc`）然後要求 Gemini CLI 優化或排除故障。例如，「我的 PATH 沒有選取 Go 二進位檔案，您能編輯我的 `.bashrc` 來修復它嗎？」AI 可以插入正確的匯出行。它會在儲存更改之前向您顯示差異以供確認。
*   **診斷錯誤：** 如果您在終端機或應用程式日誌中遇到一個神秘的錯誤，您可以複製它並將其提供給 Gemini CLI。它將分析錯誤訊息並經常建議解決步驟。這類似於人們如何使用 StackOverflow 或 Google，但 AI 直接檢查您的場景。例如：「當我運行 `npm install` 時，我收到一個 EACCES 權限錯誤——我該如何修復它？」Gemini 可能會檢測到這是 `node_modules` 中的權限問題，並引導您更改目錄所有權或使用適當的節點版本管理器。
*   **在專案之外運行：** 預設情況下，如果您在沒有 `.gemini` 上下文的目錄中運行 `gemini`，這僅表示沒有載入專案特定上下文——但您仍然可以完全使用 CLI。這對於系統故障排除等臨時任務非常有用。您可能沒有任何程式碼檔案供它考慮，但您仍然可以透過它運行 shell 命令或讓它獲取網路資訊。本質上，您將 Gemini CLI 視為一個 AI 驅動的終端機，它可以為您做事情，而不僅僅是聊天。
*   **工作站自訂：** 想要更改設定或安裝新工具？您可以詢問 Gemini CLI：「在我的系統上安裝 Docker」或「配置我的 Git 以使用 GPG 簽署提交。」CLI 將嘗試執行這些步驟。它可能會從網路獲取說明（使用搜尋工具），然後運行適當的 shell 命令。當然，始終觀察它正在做什麼並批准命令——但它可以透過自動化多步驟設定過程來節省時間。一個真實的範例：用戶要求 Gemini CLI「將我的 macOS Dock 偏好設定為自動隱藏並移除延遲」，AI 能夠執行必要的 `defaults write` 命令。

將此模式視為將 Gemini CLI 用作智慧型 shell。事實上，您可以將其與技巧 16（shell 傳遞模式）結合使用——有時您可能會進入 `!` shell 模式以驗證某些內容，然後返回 AI 模式以讓它分析輸出。

**警告：** 在執行系統級任務時，請謹慎使用具有廣泛影響的命令（例如 `rm -rf /` 或系統配置更改）。Gemini CLI 通常會要求確認，並且在您看到它之前不會運行任何內容。但作為高級用戶，您應該了解正在進行哪些更改。如果不確定，請在運行之前要求 Gemini 解釋命令（例如，「解釋 `defaults write com.apple.dock autohide-delay -float 0` 的作用」——如果您以這種方式提示它，它會樂意解釋而不是直接執行）。

**故障排除獎勵：** 另一個巧妙的用途是使用 Gemini CLI 解析日誌或設定檔以查找問題。例如，「掃描此 Apache 設定以查找錯誤」（帶有 `@httpd.conf`），或「查看 syslog 以查找昨天下午 2 點左右的錯誤」（帶有 `@/var/log/syslog`，如果可存取）。這就像有一個共同管理員。它甚至可以建議崩潰的可能原因或為常見錯誤模式提出修復方案。

總之，請毫不猶豫地將 Gemini CLI 作為您處理環境問題的助手。它旨在加速您的所有工作流程——不僅僅是編寫程式碼，還包括維護您編寫程式碼的系統。許多用戶報告說，在 Gemini 的幫助下自訂他們的開發環境，感覺就像有一個技術夥伴隨時待命，處理繁瑣或複雜的設定步驟。

## 技巧 10：YOLO 模式 - 自動批准工具操作（謹慎使用）

**快速用例：** 如果您感到自信（或喜歡冒險），您可以讓 Gemini CLI 運行工具操作，而無需每次都徵求您的確認。這就是 YOLO 模式（You Only Live Once）。它透過 `--yolo` 標誌或在會話期間按下 `Ctrl+Y` 啟用。在 YOLO 模式下，一旦 AI 決定使用工具（例如運行 shell 命令或寫入檔案），它會立即執行，而無需「批准？(y/n)」提示。

**為什麼使用 YOLO 模式？** 主要是在您信任 AI 的操作時，為了速度和便利。經驗豐富的用戶可能會在執行大量重複的安全操作時開啟 YOLO。例如，如果您要求 Gemini 一個接一個地生成 10 個不同的檔案，每次都批准會減慢流程；YOLO 模式會讓它們全部自動寫入。另一種情況是在完全自動化的腳本或 CI 管道中使用 Gemini CLI——您可能會使用 `--yolo` 無頭運行它，這樣它就不會暫停以進行確認。

要從一開始就以 YOLO 模式啟動，請使用以下命令啟動 CLI：

```pwsh
gemini --yolo
```

或簡寫形式 `gemini -y`。您會在 CLI 中看到一些指示（例如不同的提示或通知），表明自動批准已開啟。在互動式會話期間，您可以隨時按下 `Ctrl+Y` 來切換它——CLI 通常會在頁腳顯示類似「YOLO 模式已啟用（所有操作自動批准）」的訊息。

**重大警告：** YOLO 模式功能強大但有風險。Gemini 團隊自己將其標記為「大膽的用戶」——這意味著您應該意識到 AI 可能會在未經詢問的情況下執行危險命令。在正常模式下，如果 AI 決定運行 `rm -rf /`（最壞的情況），您顯然會拒絕。在 YOLO 模式下，該命令將立即運行（並可能毀了您的一天）。雖然這種極端錯誤不太可能發生（AI 的系統提示包含安全指南），但確認的全部意義在於捕獲任何不必要的動作。YOLO 移除了該安全網。

**YOLO 的最佳實踐：** 如果您想要一些便利而沒有完全的風險，請考慮允許特定命令。例如，您可以在設定中配置某些工具或命令模式不需要確認（例如允許所有 git 命令或只讀操作）。事實上，Gemini CLI 支援配置以跳過特定命令的確認：例如，您可以設定類似 `“tools.shell.autoApprove”: [”git “, “npm test”]` 的內容以始終運行這些命令。這樣，您可能不需要全域 YOLO 模式——您只選擇性地 YOLO 安全命令。另一種方法：在使用 YOLO 時在沙盒或容器中運行 Gemini，這樣即使它做了什麼瘋狂的事情，您的系統也會受到保護（Gemini 有一個 `--sandbox` 標誌可以在 Docker 容器中運行工具）。

許多高級用戶經常開啟和關閉 YOLO——在進行一系列次要檔案編輯或查詢時開啟它，在即將執行關鍵操作時關閉它。您也可以這樣做，使用鍵盤快捷鍵作為快速切換。

總之，YOLO 模式以犧牲監督為代價消除了摩擦。這是一個專業功能，應謹慎和明智地使用。它真正證明了對 AI 的信任（或魯莽！）。如果您是 Gemini CLI 的新手，您應該避免使用 YOLO，直到您清楚了解它傾向於做什麼的模式。如果您使用它，請加倍使用版本控制或備份——以防萬一。

（如果這有任何安慰，您並不孤單——社區中的許多人都在開玩笑說「我 YOLO 了，Gemini 做了些瘋狂的事情。」所以使用它，但是……嗯，您只活一次。）
