## 技巧 17：將每個 CLI 工具視為潛在的 GEMINI 工具

**快速用例：** 意識到 Gemini CLI 可以利用您系統上安裝的任何命令列工具作為其解決問題的一部分。AI 可以存取 shell，因此如果您有 `cURL`、`ImageMagick`、`git`、`Docker` 或任何其他工具，Gemini 可以在適當時調用它。換句話說，您的整個 `$PATH` 都是 AI 的工具包。這大大擴展了它可以做的事情——遠遠超出其內建工具。

例如，假設您詢問：「將此資料夾中的所有 PNG 圖像轉換為 WebP 格式。」如果您安裝了 ImageMagick 的 `convert` 工具，Gemini CLI 可能會計畫類似於：對每個檔案使用帶有 `convert` 命令的 shell 迴圈。事實上，早期部落格中的一個範例就展示了這一點，用戶提示批次轉換圖像，Gemini 執行了一個帶有 `convert` 工具的 shell 一行程式碼。

另一個場景：「將我的應用程式部署到 Docker。」如果 Docker CLI 存在，AI 可以根據需要調用 `docker build` 和 `docker run` 步驟。或者「使用 FFmpeg 從 `video.mp4` 中提取音訊」——它可以建構 `ffmpeg` 命令。

這個技巧是關於心態的：Gemini 不受其程式碼（已經很廣泛）的限制。它可以找出如何使用其他可用程式來實現目標。它知道常見的語法，並且可以在需要時閱讀幫助文字（它可以對工具調用 `--help`）。唯一的限制是安全性：預設情況下，它會要求確認它提出的任何 `run_shell_command`。但是，當您感到舒適時，您可能會自動允許某些良性命令（請參閱 YOLO 或 allowed-tools 配置）。

**注意環境：** 「能力越大，責任越大。」由於每個 shell 工具都是公平的，您應該確保您的 `$PATH` 不包含任何您不希望 AI 無意中運行的內容。這就是技巧 19（自訂 PATH）的用武之地——一些用戶為 Gemini 建立了一個受限制的 `$PATH`，這樣它就不能，例如，直接調用系統破壞性命令，或者可能不會遞歸調用 gemini（以避免迴圈）。重點是，預設情況下，如果 `gcc` 或 `terraform` 或任何東西在 `$PATH` 中，Gemini 都可以調用它。這並不意味著它會隨機這樣做——只有在任務需要時——但這是可能的。

**思維列車範例：** 想像您詢問 Gemini CLI：「設定一個基本的 HTTP 伺服器，提供目前目錄。」AI 可能會想：「我可以使用 Python 的內建伺服器來實現。」然後它發出 `!python3 -m http.server 8000`。現在它只是使用了一個系統工具 (Python) 來啟動伺服器。這是一個無害的範例。另一個：「檢查此 Linux 系統上的記憶體使用情況。」AI 可能會使用 `free -h` 命令或從 `/proc/meminfo` 讀取。它實際上正在做系統管理員會做的事情，透過使用可用的命令。

**所有工具都是 AI 的擴展：** 這有點未來主義，但請考慮任何命令列程式都可以被視為 AI 可以調用以擴展其功能的「函數」。需要解決數學問題？它可以調用 `bc`（計算器）。需要操作圖像？它可以調用圖像處理工具。需要查詢資料庫？如果 CLI 用戶端已安裝且憑證存在，它可以使用它。可能性是廣闊的。在其他 AI 代理框架中，這被稱為工具使用，而且 Gemini CLI 的設計對其代理決定正確工具的能力充滿信任。

**出錯時：** 另一方面是如果 AI 誤解了工具或對工具產生幻覺。它可能會嘗試調用不存在的命令，或使用錯誤的標誌，導致錯誤。這不是什麼大問題——您會看到錯誤並可以糾正或澄清。事實上，Gemini CLI 的系統提示可能會引導它首先進行試運行（只是提出命令）而不是盲目執行。因此，您通常有機會捕獲這些錯誤。隨著時間的推移，開發人員正在改進工具選擇邏輯以減少這些失誤。

主要 takeaway 是將 Gemini CLI 視為擁有一把非常大的瑞士軍刀——不僅僅是內建刀片，還有您作業系統中的每個工具。如果它是標準的，您不必指導它如何使用它們；通常它知道或可以找出。這顯著擴大了您可以完成的事情。這就像有一個初級開發人員或 devops 工程師，他知道如何運行您安裝的幾乎任何程式。

作為專業用戶，您甚至可以安裝額外的 CLI 工具，專門賦予 Gemini 更多功能。例如，如果您為雲端服務（AWS CLI、GCloud CLI 等）安裝 CLI，理論上 Gemini 可以在提示時利用它來管理雲端資源。始終確保您理解並信任運行的命令，特別是對於強大的工具（您不會希望它意外啟動巨大的雲端實例）。但如果使用得當，這個概念——一切都是 Gemini 工具——正是它在您將其整合到您的環境中時，使其能力呈指數級增長的原因。

## 技巧 18：利用多模態 AI - 讓 GEMINI 看到圖像等

**快速用例：** Gemini CLI 不僅限於文字——它是多模態的。這意味著它可以分析圖像、圖表，甚至 PDF。利用這一點。例如，您可以說：「這是一個錯誤對話框的螢幕截圖，`@./error.png`——幫我解決這個問題。」AI 將「看到」圖像並相應地回應。

Google Gemini 模型（及其前身 Codey 形式的 PaLM2）的突出功能之一是圖像理解。在 Gemini CLI 中，如果您使用 `@` 引用圖像，模型會收到圖像資料。它可以輸出描述、分類或對圖像內容進行推理。我們已經討論過按內容重新命名圖像（技巧 14）和描述螢幕截圖（技巧 7）。但讓我們考慮其他創意用途：

*   **UI/UX 回饋：** 如果您是與設計師合作的開發人員，您可以放置 UI 圖像並要求 Gemini 提供回饋或生成程式碼。「查看此 UI 模型 `@mockup.png` 並為其產生 React 元件結構。」它可以識別圖像中的元素（標題、按鈕等）並概述程式碼。
*   **組織圖像：** 除了重新命名之外，您可能有一個混合圖像的資料夾，並希望按主題（例如，日落、山脈、人物）排序到子資料夾中。「將 `./photos/` 中的圖像按主題（例如，日落、山脈、人物）排序到子資料夾中。」AI 可以查看每張照片並對其進行分類（這類似於某些照片應用程式使用 AI 所做的事情——現在您可以透過 Gemini 使用自己的腳本來完成）。
*   **OCR 和資料提取：** 如果您有錯誤文字的螢幕截圖或文件的照片，Gemini 通常可以從中讀取文字。例如，「從 `invoice.png` 中提取文字並將其放入結構化格式。」如 Google Cloud 部落格範例所示，Gemini CLI 可以處理一組發票圖像並輸出其資訊的表格。它基本上執行了 OCR + 理解以從發票圖片中獲取發票號碼、日期、金額。這是一個高級用例，但使用底層的多模態模型完全可能。
*   **理解圖形或圖表：** 如果您有圖形螢幕截圖，您可以詢問「解釋此圖表的關鍵見解 `@chart.png`。」它可能會解釋軸和趨勢。準確性可能有所不同，但這是一個巧妙的嘗試。

為了使其切實可行：當您 `@image.png` 時，請確保圖像不要太大（儘管模型可以處理相當大的圖像）。CLI 可能會對其進行編碼並將其發送到模型。回應可能包括描述或進一步的操作。您也可以在一個提示中混合文字和圖像引用。

**非圖像模態：** CLI 和模型也可能透過工具處理 PDF 和音訊。例如，如果您 `@report.pdf`，Gemini CLI 可能會使用 PDF 到文字工具在底層提取文字，然後進行總結。如果您 `@audio.mp3` 並要求轉錄，它可能會使用音訊到文字工具（例如語音識別功能）。備忘單建議支援引用 PDF、音訊、視訊檔案，大概是透過調用適當的內部工具或 API。因此，「轉錄此訪談音訊：`@interview.wav`」實際上可能有效（如果現在不行，可能很快就會，因為底層的 Google 語音轉文字 API 可以插入）。

**豐富的輸出：** 多模態也意味著 AI 可以在整合後在回應中返回圖像（儘管在 CLI 中它通常不會直接顯示它們，但它可以儲存圖像檔案或輸出 ASCII 藝術等）。MCP 功能提到工具可以返回圖像。例如，AI 繪圖工具可以生成圖像，Gemini CLI 可以呈現它（可能透過開啟它或提供連結）。

**重要：** CLI 本身是基於文字的，因此您不會在終端機中看到圖像（除非它能夠進行 ASCII 預覽）。您只會得到分析。因此，這主要是關於讀取圖像，而不是顯示它們。如果您在 VS Code 整合中，它可能會在聊天視圖中顯示圖像。

總之，在使用 Gemini CLI 時，不要忘記 GUI 中的「I」——它在許多情況下可以像處理文字一樣處理視覺。這開啟了視覺偵錯、設計協助、從螢幕截圖中提取資料等工作流程，所有這些都在同一個工具下。這是其他一些 CLI 工具可能尚未具備的差異化因素。隨著模型的改進，這種多模態支援只會變得更加強大，因此這是一項值得利用的未來證明技能。
